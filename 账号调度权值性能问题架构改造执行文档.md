# 账号调度权值性能问题架构改造执行文档

## 目标

在账号管理列表中继续展示管理员需要的 OpenAI 调度权值，但彻底移除列表接口中的全量候选池计算。改造完成后，`GET /api/v1/admin/accounts` 的调度权值展示必须只读取当前分页账号相关的调度权值投影，时间复杂度与当前页账号数相关，而不是与全站 OpenAI 账号数相关。

本方案不是隐藏列、减少默认字段、加超时这种止血方案。必须把调度权值从账号列表接口的同步派生计算，移动到调度系统拥有的读模型中。

## 范围与阶段边界

本文档分为两个阶段，但默认施工范围是**第一阶段：账号列表调度权值读模型改造**。

1. 第一阶段只解决 `GET /api/v1/admin/accounts` 展示 `scheduler_score` 时破坏分页边界的问题。
2. 第一阶段不得修改 OpenAI 网关真实账号选择器的选号语义，不得把展示用 `scheduler_score:v1` 读模型用于真实调度。
3. `GetActiveAccountLoadMap` 是第一阶段和第二阶段共享的底层能力。第一阶段只允许在后台重建展示读模型时使用它；真实调度器是否使用 active load map，必须进入第二阶段并由开关控制。
4. 第二阶段是独立架构改造，必须在第一阶段完成并验收后单独评审、单独提交、单独灰度。除非负责人明确批准，不允许把第二阶段和第一阶段混入同一个上线批次。
5. 本文档中第二阶段章节用于约束后续方向，不是第一阶段执行者的默认交付范围。

## 当前根因

当前代码在账号列表接口中为了展示 `scheduler_score` 破坏了分页边界。

关键路径如下：

1. 前端 `scheduler_score` 列已在 `DEFAULT_HIDDEN_COLUMNS` 中默认隐藏，但用户手动勾选列可见后，后端才会触发昂贵的全量候选池计算。
   - `frontend/src/views/admin/AccountsView.vue`
   - `DEFAULT_HIDDEN_COLUMNS` **已经包含** `scheduler_score`（当前代码状态）。
   - 前端同时有存量迁移逻辑自动将旧布局中的 `scheduler_score` 强制隐藏（`HIDDEN_COLUMNS_VERSION_KEY`）。
   - 后端通过 `include_scheduler_score` 查询参数门控，默认 `false`。
   - 因此问题发生条件是：管理员主动勾选该列可见 → 前端发送 `include_scheduler_score=1` → 后端触发全量计算。
   - **本文档的目标不是把列继续隐藏，而是让列可见时也能快速返回。**

2. 后端列表接口在分页查询之后，又为当前页 OpenAI 账号同步构造完整候选池。
   - `backend/internal/handler/admin/account_handler.go`
   - `AccountHandler.List`
   - `listAccountSchedulerScoreFilterPool`
   - `buildOpenAIAccountSchedulerScores`

3. `listAccountSchedulerScoreFilterPool` 调用 `AdminService.ListAccountsForSchedulerScoreFilter`，该方法绕过分页读取所有符合筛选条件的 OpenAI 账号。
   - `backend/internal/service/admin_service.go`
   - `backend/internal/repository/account_repo.go`
   - `ListAllWithFilters`

4. `buildOpenAIAccountSchedulerScores` 对当前页出现的每个分组继续调用 `ListOpenAISchedulableAccountsForSchedulerScore`，读取对应分组的完整可调度账号池。

5. `fetchOpenAIAccountLoadMap` 对完整候选账号并集调用 `ConcurrencyService.GetAccountsLoadBatch`。
   - `backend/internal/service/concurrency_service.go`
   - `backend/internal/repository/concurrency_cache.go`

6. Redis 实现中 `GetAccountsLoadBatch` 对每个账号至少发出 `ZREMRANGEBYSCORE`、`ZCARD`、`GET` 三类命令。168831 个账号会导致单个账号列表请求发起约 50 万级 Redis 命令。

结论：慢的根因不是分页失效，也不是前端渲染慢，而是账号列表接口把调度器的全量候选池打分逻辑同步塞进了列表热路径。

## 不允许做的事

执行者必须严格遵守以下禁区：

1. 不允许只把 `scheduler_score` 默认隐藏作为最终方案。
2. 不允许在 `AccountHandler.List` 中继续调用 `ListAccountsForSchedulerScoreFilter`。
3. 不允许在 `AccountHandler.List` 中继续调用 `ListOpenAISchedulableAccountsForSchedulerScore`。
4. 不允许在 `AccountHandler.List` 中为了展示调度权值调用 `GetAccountsLoadBatch` 读取全量候选池负载。
5. 不允许把调度分数定义为“当前 UI 筛选条件下的相对分数”。最终语义必须是“真实调度桶中的调度分数”。
6. 不允许为了这个展示功能修改 OpenAI 网关真实选号逻辑。
7. 不允许用 `ListAllWithFilters`、全表扫描、全量 Redis pipeline 作为列表接口的兜底。
8. 不允许引入新的数据库迁移作为第一选择。读模型先使用 Redis，因为现有调度快照已经使用 Redis。
9. 不允许删除和本任务无关的代码、格式化无关文件、重构无关模块。
10. 不允许在 nil 判断中使用 Go 的 `interface` 值比较陷阱。
    - `NewSchedulerScoreReadModel` 在 `rdb == nil` 时必须**显式返回一个 no-op 实现的结构体指针**，而非返回 `nil`。
    - 因为 Go 中 `(*noopReadModel)(nil)` 赋值给 `SchedulerScoreReadModel` 接口后，接口值不等于 `nil`，会导致 `if readModel == nil` 判断失效。
    - 必须定义 `type noopReadModel struct{}`，`NewSchedulerScoreReadModel` 在 `rdb == nil` 时返回 `&noopReadModel{}`。Service 层不依赖接口值是否为 nil 判断 Redis 是否可用。

## 目标架构

新增一个调度权值读模型，由调度系统负责异步生成，由账号列表接口按当前页读取。

核心原则：

1. 调度权值属于调度域，不属于账号列表域。
2. 分数按已有 `service.SchedulerBucket` 归属，不按账号列表筛选条件归属。
3. 写入复杂度允许是 `O(bucket_size)`，但只能发生在调度快照重建、账号变更、分组变更、配置变更等后台路径。
4. 账号列表读取复杂度必须是 `O(page_size * account_bucket_count)`。
5. 账号列表展示缺失投影时必须优雅降级为空或标记 missing/stale，不能触发同步全量重算。

已有调度桶定义在：

```go
// backend/internal/service/scheduler_cache.go
type SchedulerBucket struct {
    GroupID  int64
    Platform string
    Mode     string
}
```

必须复用这个类型。

### 调度桶归一化规则

读模型写入和账号列表读取必须使用与 `SchedulerSnapshotService` 一致的 bucket 归一化规则，不能在账号列表侧直接用账号分组拼接 Redis key。

第一阶段规则：

1. 只生成和读取 `SchedulerBucket{Platform: PlatformOpenAI, Mode: SchedulerModeSingle}`。
2. `forced` / `mixed` bucket 不参与账号列表调度权值展示；如果重建路径遇到这些 bucket，直接跳过或删除对应 score read model。
3. standard run mode：
   - 未分组账号读取 `GroupID=0`。
   - 已分组账号读取账号所属分组的 `GroupID`。
4. simple run mode：
   - 所有账号统一读取 `GroupID=0`，必须与 `SchedulerSnapshotService.normalizeGroupID` 的行为一致。
5. 如果未来 OpenAI 兼容平台需要纳入展示，必须先定义平台归一化规则；第一阶段只处理 `PlatformOpenAI`。

`SchedulerScoreService` 必须持有 `*config.Config` 或等价的 bucket resolver，用于判断 run mode。`cfg == nil` 时按 standard run mode 处理。

## 分数语义

第一阶段只实现 OpenAI 账号调度权值读模型。

对于账号列表中的 OpenAI 账号：

1. 未分组账号读取桶：
   - `SchedulerBucket{GroupID: 0, Platform: service.PlatformOpenAI, Mode: service.SchedulerModeSingle}`

2. 已分组账号在 standard run mode 下读取每个所属分组桶：
   - `SchedulerBucket{GroupID: groupID, Platform: service.PlatformOpenAI, Mode: service.SchedulerModeSingle}`

3. simple run mode 下，即使账号绑定了分组，也读取 group 0 single bucket。

4. 如果账号存在 forced 路由展示需求，另行在诊断能力中展示 forced 桶，不要混入账号列表默认列。

5. `scheduler_score` 字段保留为兼容字段：
   - 未分组账号：来自未分组桶。
   - 已分组账号：取 `scheduler_scores` 中优先展示的一条。优先级规则为：`group_priority` 非空优先；非空时数值更小者优先；相同或都为空则 `base_score` 更大者优先；再相同按 `group_id` 升序。

6. `scheduler_scores` 字段表示账号所在真实调度桶中的每个分组分数。

前端提示文案必须从“当前筛选条件限定的候选账号”改为“真实调度桶候选账号”。

```go
package service

import (
    "context"
    "time"

    "github.com/Wei-Shaw/sub2api/internal/config"
)

type SchedulerScoreSnapshot struct {
    AccountID              int64           `json:"account_id"`

    GroupID                int64           `json:"group_id"`
    GroupName              string          `json:"group_name,omitempty"`
    GroupPriority          *int            `json:"group_priority,omitempty"`
    BaseScore              float64         `json:"base_score"`
    StickyScore            float64         `json:"sticky_score"`
    StickyScoreInfinity    bool            `json:"sticky_score_infinity"`
    StickyWeightedEnabled  bool            `json:"sticky_weighted_enabled"`
    Rank                   int             `json:"rank"`
    BucketSize             int             `json:"bucket_size"`
    UpdatedAt              time.Time       `json:"updated_at"`
    Version                int64           `json:"version"`
}

type SchedulerScoreReadModel interface {
    NextVersion(ctx context.Context, bucket SchedulerBucket) (int64, error)
    GetScores(ctx context.Context, bucket SchedulerBucket, accountIDs []int64) (map[int64]SchedulerScoreSnapshot, error)
    SetBucketScores(ctx context.Context, bucket SchedulerBucket, scores []SchedulerScoreSnapshot, version int64, updatedAt time.Time) error
    DeleteBucket(ctx context.Context, bucket SchedulerBucket) error
}
```

再定义服务类型：

```go
type SchedulerScoreService struct {
    readModel          SchedulerScoreReadModel
    concurrencyService *ConcurrencyService
    rateLimitService   *RateLimitService
    cfg                *config.Config
}
```

构造函数：

```go
func NewSchedulerScoreService(
    readModel SchedulerScoreReadModel,
    concurrencyService *ConcurrencyService,
    rateLimitService *RateLimitService,
```

必须允许依赖为 nil。nil 时行为如下：

4. `cfg == nil`：按 standard run mode 处理 bucket 归一化。

## Redis 读模型

新增文件：

```text
backend/internal/repository/scheduler_score_read_model.go
```

实现：

```go
type schedulerScoreReadModel struct {
    rdb *redis.Client
}

func NewSchedulerScoreReadModel(rdb *redis.Client) service.SchedulerScoreReadModel
```

如果 `rdb == nil`，必须返回 no-op 实现，不能返回 nil。

Redis key 规则必须集中在同一个文件的私有函数中，不允许散落字符串拼接。

第一阶段必须使用 active_version 方案；以下非版本化 key 只用于说明字段形态，不允许作为最终实现：

```text
scheduler_score:v1:{bucket:<bucket.String()>}:scores
scheduler_score:v1:{bucket:<bucket.String()>}:meta
```

其中 `scores` 是 Redis Hash：

```text
field = account_id 十进制字符串
value = SchedulerScoreSnapshot JSON
```

其中 `meta` 是 Redis Hash：

```text
version
updated_at
bucket_size
```

实现要求：

1. `NextVersion` 使用 bucket 维度 `INCR`，不得使用本机时钟生成 active 版本。
2. `GetScores` 使用 `HMGET`，只读取传入的 `accountIDs`。
3. `GetScores` 必须去重 accountIDs，保持返回 map。
4. `SetBucketScores` 必须分块写入，块大小常量：
   ```go
   const schedulerScoreRedisChunkSize = 1000
   ```
5. `SetBucketScores` 不允许构造十万级单次 pipeline。
6. `SetBucketScores` 必须先写入未激活的版本化 scores/meta key，再通过 active_version 原子切换，避免读到半成品。
7. active、scores、meta、version、旧版本 key 必须使用同一 hash tag，保证 Redis Cluster 下 Lua 脚本不跨 slot。
8. 不允许 delete old 后慢慢 hset new，也不允许用非版本化 key 直接覆盖。
9. `DeleteBucket` 删除 `active` key 和当前 active version 对应的 scores/meta；历史版本如果已设置 TTL，可以让其自然过期，不允许用 `SCAN scheduler_score:v1:*` 作为请求路径兜底。
10. JSON 反序列化失败时忽略该 account 的单条结果，并记录 warn 日志，不允许让整个列表失败。

active_version 方案的 key 必须是：

```text
scheduler_score:v1:{bucket:<bucket.String()>}:version
scheduler_score:v1:{bucket:<bucket.String()>}:active
scheduler_score:v1:{bucket:<bucket.String()>}:scores:<version>
scheduler_score:v1:{bucket:<bucket.String()>}:meta:<version>
```

版本规则：

1. `Version` 不允许使用调用方本机时钟或 `time.Now().UnixNano()` 直接生成。
2. Redis 实现的 `NextVersion` 必须使用同 bucket 的 `version` key 执行 `INCR`，得到单调递增版本号。
3. no-op 实现的 `NextVersion` 可以返回 `0, nil`；因为 no-op 不会真实写入 active version。
4. Redis 实现的 `SetBucketScores` 收到的 `version` 必须来自 `NextVersion`。如果 `version <= 0`，必须返回错误，不允许写入；no-op 实现可以直接返回 nil。
5. active 切换必须防止版本回滚：原子切换脚本只允许 `new_version >= current_active_version` 时更新 active；否则丢弃本次切换，并给新版本设置短 TTL。

读取流程：

1. `GET active`
2. 如果为空，返回空 map。
3. `HMGET scores:<version> accountIDs...`

写入流程：

1. `SchedulerScoreService.RebuildBucketScores` 在调用 `SetBucketScores` 前先调用 `NextVersion(ctx, bucket)` 分配单调版本；同一 bucket 内所有 `SchedulerScoreSnapshot.Version` 都使用该版本。
2. `GET active` 读取旧 active version。
3. `HSET scores:<version> ...`
4. `HSET meta:<version> ...`
5. 给新版本 scores/meta 设置临时 TTL（推荐 10 分钟），防止切换 active 前进程崩溃留下永不过期的未激活版本。
6. 使用同 slot 的 Lua 脚本原子执行：
   - `PERSIST scores:<version>`
   - `PERSIST meta:<version>`
   - 当 `active` 为空或 `version >= active` 时，`SET active <version>`
   - 给旧版本 scores/meta 设置 TTL（推荐 300 秒）
   - 如果 `version < active`，不给 active 回滚，并给本次 scores/meta 保持短 TTL 后返回
7. **禁止**先 `SET active <version>` 再单独 `PERSIST` 新版本。否则进程在两步之间崩溃时，`active` 会指向一个仍带临时 TTL 的版本，TTL 到期后列表会长期读空。
8. active、scores、meta、version、旧版本 key 都必须使用同一 hash tag；Redis Cluster 环境下跨 slot 事务不可接受。
9. **崩溃恢复保障**：旧版本 key 不应立即删除，而是设置一个 TTL 作为兜底。
   - 原因：如果新版本写入完成后、`SET active` 之前进程崩溃，新版本数据已写入但 `active` 仍指向旧版本或空。此时旧版本数据是唯一可用的完整数据。
   - 做法：写入新版本成功、原子切换 active 成功后，再让旧版本 key 进入 300 秒 TTL。后台清理如需 `UNLINK`，必须在 TTL 之后执行，不能和 active 切换同事务立即删除旧版本。
   - 读取侧：`GET active` 返回的版本如果 HMGET 返回空（极少情况下旧版本已被清理），回退返回空 map，不允许重试其他版本。

必须使用 active_version 方案，因为它能同时满足 Redis Cluster 同 slot、崩溃恢复和并发重建不回滚 active 的要求。

### 运维考量：Redis 内存估算与清理

`SchedulerScoreSnapshot` 的真实内存成本必须按“bucket 成员关系总数”估算，而不是只按账号数估算。

估算公式：

```text
score_rows = sum(bucket_size)
memory ≈ score_rows × (json_size + redis_hash_overhead) × active_versions
```

说明：

- 如果 10 万个 OpenAI 账号互斥分布在 50 个 bucket 中，`score_rows` 约等于 10 万。
- 如果账号可属于多个分组，`score_rows` 等于账号分组成员关系总数，可能大于账号数。
- JSON 字段名、Redis Hash 元数据、对象分配和旧版本 TTL 都会带来额外开销，不能只按 200-300 字节估算。
- `active_version` 方案通常同时存在 1-2 个版本（当前 active + 新版本写入中或旧版本 TTL 兜底）。
- 第一阶段仅 OpenAI 平台，但仍应按线上 bucket 成员数做一次 `MEMORY USAGE` 抽样确认。

建议在运维监控中增加 `scheduler_score:v1:*` 的 key 数量和内存占用告警。如需清理，使用：

```bash
redis-cli --scan --pattern "scheduler_score:v1:*" | xargs redis-cli UNLINK
```

### ConcurrencyService 新增方法

```go
func (s *ConcurrencyService) GetActiveAccountLoadMap(ctx context.Context) (map[int64]*AccountLoadInfo, error)
```

行为要求：

1. `s == nil` 或 `s.cache == nil` 时返回空 map、nil error。
2. 委托底层 `s.cache.GetActiveAccountLoadMap(ctx)` 执行裸 Redis 读取。
3. **缓存、singleflight、超时控制必须在 `ConcurrencyService` 层实现**，参考现有 `getAccountsLoadBatch` 模式（`concurrency_service.go:413-468`）：
   - 新增 `activeLoadGroup singleflight.Group` 字段防止并发请求重复回源。

   - 超时控制 3 秒（`context.WithTimeout`）。
4. 不允许要求调用方传入全量账号列表。
5. 返回 map 中必须填充 `AccountID`、`CurrentConcurrency`、`WaitingCount`。
6. `LoadRate` 填 0；调用方必须按已知账号 `EffectiveLoadFactor()` 重算。
7. Redis 失败时返回错误，由调用方决定降级为空负载。

### ConcurrencyCache Redis 实现要求

底层 `ConcurrencyCache.GetActiveAccountLoadMap` 的 Redis 实现：

1. 使用 Redis `TIME` 获取 `now`，再用 `ZRANGEBYSCORE concurrency:account:active_index now +inf LIMIT offset chunk` 分块读取仍处于活跃窗口的账号 ID；不允许使用 `ZRANGE 0 -1` 全量读取过期 member。
2. 分页读取期间不要同步删除 active index member。必须先完成本次 range 分页，再批量删除 stale/非法 member，避免 `LIMIT offset` 因集合收缩而跳过账号。
3. 分页变量要求：
   - `offset` 初始为 0。
   - 每次读取 `Count=activeAccountLoadReadChunkSize`。
   - 每次读取后 `offset += len(rawMembers)`。
   - 当 `len(rawMembers) < activeAccountLoadReadChunkSize` 时结束。
   - 结果 map 按 accountID 去重；并发写入导致的重复 member 不能影响返回正确性。
4. 每个分块内对活跃账号 ID 调用 pipeline 读取 `ZREMRANGEBYSCORE`（槽位过期清理）、`ZCARD`（槽位数）和 `GET`（等待计数）。
5. 解析失败的 member 必须跳过并记录 warn，并加入待删除列表。
6. 读取真实负载后，如果 `slotCount == 0 && waitCount <= 0`，加入待删除列表；本次分页完成后再从 active index 删除，避免索引残留。
7. 对每个仍有真实负载的活跃账号构造 `AccountLoadInfo`。
8. 必须设置合理的超时（推荐 3 秒），避免阻塞调用方。
9. 如果 active index 为空，返回空 map，不是 error。
10. 返回的 `AccountLoadInfo.LoadRate` 字段填 0；由 `ConcurrencyService` 层的调用方按 `EffectiveLoadFactor()` 重新计算。
11. **禁止在 `ConcurrencyCache` 实现中添加缓存、singleflight 等横切逻辑**——这些属于 `ConcurrencyService` 层职责。

## 调度权值重建算法

在 `SchedulerScoreService` 中实现：

```go
func (s *SchedulerScoreService) RebuildBucketScores(ctx context.Context, bucket SchedulerBucket, accounts []Account) error
```

行为必须如下：

1. 如果 `bucket.Platform != PlatformOpenAI` 或 `bucket.Mode != SchedulerModeSingle`，调用 `DeleteBucket(ctx, bucket)` 或直接返回 nil。第一阶段只为 OpenAI single bucket 生成分数，不为 forced/mixed bucket 写展示读模型。
2. 如果 `len(accounts) == 0`，调用 `DeleteBucket(ctx, bucket)`，然后返回 nil。
3. 获取账号负载时，只允许调用：
   ```go
   s.concurrencyService.GetActiveAccountLoadMap(ctx)
   ```
   不允许调用 `GetAccountsLoadBatch`。
4. **负载读取失败保护**：如果 `concurrencyService != nil` 且 `GetActiveAccountLoadMap` 返回错误，必须记录 warn 并跳过本次 score active version 发布，保留旧版本；不得按空负载发布新版本。
   - 理由：列表展示读模型是派生数据，Redis 负载读取异常时宁可显示旧值或缺失值，也不要发布一版“全员零负载”的误导性分数。
   - 行为：`RebuildBucketScores` 返回错误，由 `SchedulerSnapshotService.rebuildBucket` 记录 `scheduler_score_rebuild_failed`，但不能让调度快照重建失败。
5. **空 active map 保护**：如果 `concurrencyService != nil`、`GetActiveAccountLoadMap` 成功但返回空 map 且 `len(accounts) > 100`，必须记录 warn 日志 `active_load_map_empty_for_large_bucket`，并按以下规则处理：
   - 如果这是自动后台重建，默认跳过本次 score active version 发布，保留旧版本；没有旧版本时账号列表显示 missing。
   - 如果运维确认系统确实无活跃请求，需要首次生成展示读模型，可通过诊断接口或一次性运维任务显式 force rebuild，但该 force 行为不能出现在账号列表请求路径。
   - 恢复：调度快照的周期性全量重建会在 active index 恢复后自动生成新版本。
6. 对 bucket 内账号构造 `[]*Account`，调用现有分数构造函数：
   - 有 `RateLimitService`：`rateLimitService.BuildOpenAIAccountSchedulerScoreSnapshot(ctx, openAIAccounts, activeLoadMap)`
   - 无 `RateLimitService`：`BuildOpenAIAccountSchedulerScoreSnapshot(openAIAccounts, activeLoadMap)`
   - **注意**：两种路径的 `StickyWeightedEnabled` 行为不同。有 `RateLimitService` 时，`StickyWeightedEnabled` 从网关配置读取（`isOpenAIAdvancedSchedulerStickyWeightedEnabled`）；无 `RateLimitService` 时，`BuildOpenAIAccountSchedulerScoreSnapshot` 硬编码 `StickyWeightedEnabled=false`、`StickyScoreInfinity=true`。这意味无 `RateLimitService` 的部署环境（如测试环境）生成的分数中粘性加权始终关闭，与生产环境行为可能不一致。这是可接受的，因为读模型仅用于后台展示，不参与真实调度。
7. `activeLoadMap` 只包含活跃账号负载。缺失账号必须视为 0 负载，不允许补查。
   - `GetActiveAccountLoadMap` 返回的是“活跃账号的当前并发数和等待数”。调用方拿到 bucket 内账号列表后，必须按 `account.EffectiveLoadFactor()` 重新计算 `LoadRate`：
     ```go
     loadRate := 0
     if maxConcurrency > 0 {
         loadRate = (currentConcurrency + waitingCount) * 100 / maxConcurrency
     }
     ```
   - 不允许直接信任 active load map 中可能缺省或陈旧的 `LoadRate` 字段。
8. 生成 `SchedulerScoreSnapshot` 时必须填充：
   - `AccountID`
   - `Bucket`
   - `GroupID`
   - `GroupName`
   - `GroupPriority`
   - `BaseScore`
   - `StickyScore`
   - `StickyScoreInfinity`
   - `StickyWeightedEnabled`
   - `Rank`
   - `BucketSize`
   - `UpdatedAt`
   - `Version`
9. `UpdatedAt` 对同一 bucket 内所有账号使用同一个 `time.Now().UTC()`。
10. `Version` 必须来自 `readModel.NextVersion(ctx, bucket)`，不得使用 `UpdatedAt.UnixNano()`。
11. `BucketSize` 等于本次参与该 bucket 打分的 OpenAI 账号数。

Rank 排序规则必须确定：

1. `BaseScore` 降序。
2. `StickyScoreInfinity == true` 排在 `false` 前面。
3. `StickyScore` 降序。
4. 账号在该分组内的 `GroupPriority` 升序，`nil` 排在有值之后。
5. `Account.Priority` 升序。
6. `Account.ID` 升序。

Rank 从 1 开始。

**排序语义说明**：`BaseScore` 的计算公式（见 `buildOpenAIAccountSchedulerScoreSnapshot`）中已经包含了 `weights.Priority * priorityFactor`，即 `Account.Priority` 已经通过权重因子间接影响了 `BaseScore`。因此规则 4-6 是**二级 tiebreaker**——仅在 `BaseScore`、`StickyScoreInfinity`、`StickyScore` 全部相同时才生效，用于保证排序的确定性（deterministic ordering）。

注意：本项目账号优先级遵循“数值越小优先级越高”的既有调度语义；账号分组优先级在仓库查询中同样按升序优先。读模型排序和兼容字段选择必须与这些既有顺序保持一致。

分组元数据填充规则：

1. `bucket.GroupID == 0`：
   - `GroupID = 0`
   - `GroupName = ""`
   - `GroupPriority = nil`
2. `bucket.GroupID > 0`：
   - 从 `account.AccountGroups` 中找到 `GroupID == bucket.GroupID` 的记录。
   - `GroupPriority = &accountGroup.Priority`
   - 如果 `accountGroup.Group != nil`，`GroupName = accountGroup.Group.Name`
   - 如果找不到对应 `AccountGroup`，保留 `GroupPriority = nil`，但仍写入分数，因为调度快照可能只带 `GroupIDs`。

## 账号列表读取算法

在 `SchedulerScoreService` 中实现：

```go
type AccountListSchedulerScore struct {
    AccountID              int64
    GroupID                *int64
    GroupName              string
    GroupPriority          *int
    BaseScore              float64
    StickyScore            float64
    StickyScoreInfinity    bool
    StickyWeightedEnabled  bool
    Rank                   int
    BucketSize             int
    UpdatedAt              time.Time
    Missing                bool
}

func (s *SchedulerScoreService) GetAccountListScores(ctx context.Context, accounts []Account) (map[int64]*AccountListSchedulerScore, map[int64][]AccountListSchedulerScore)
```

行为必须如下：

1. 只处理 `PlatformOpenAI` 账号。
2. 非 OpenAI 账号不返回任何分数。
3. 从当前页账号推导需要读取的 bucket，但必须先经过本文“调度桶归一化规则”：
   - 未分组账号读取 group 0 single bucket。
   - 已分组账号读取每个 group single bucket。
   - simple run mode 下，无论账号分组如何，都读取 group 0 single bucket。
4. 对每个 bucket 调用 `readModel.GetScores(ctx, bucket, accountIDsInThatBucket)`。
5. 不允许因为某个账号投影缺失而访问数据库或全量 Redis。
6. 返回两个 map：
   - `baseScores`：兼容 `scheduler_score`
   - `groupScores`：兼容 `scheduler_scores`
7. `baseScores` 的选择规则按本文“分数语义”章节执行。

该函数每次账号列表请求最多读取当前页账号涉及的 bucket。默认 page size 被后端限制，因此该函数必须稳定。

账号列表读取降级要求：

1. `GetAccountListScores` 必须使用短超时 context 包裹读模型读取，推荐：
   ```go
   const schedulerScoreListReadTimeout = 500 * time.Millisecond
   ```
2. 某个 bucket 读取失败、超时或 Redis 不可用时，只把该 bucket 的分数视为 missing，不允许让整个账号列表请求失败。
3. 读取失败最多记录 debug 或 warn 聚合日志，不能在 info 级别逐账号打印。
4. 降级路径禁止访问数据库、禁止调用 `GetAccountsLoadBatch`、禁止触发 `RebuildBucketScores`。
5. 返回结果中缺失投影应保持 `scheduler_score == nil` 或 `Missing=true`，由前端显示“计算中/尚未就绪”。

为避免管理员以后又要求在账号列表中塞全量分析，后续必须提供独立诊断能力。但第一阶段的账号列表热路径改造**不依赖诊断接口**，诊断接口可以放入后续提交或独立小阶段。

第一阶段如果时间有限，直接延期诊断接口，不要为了诊断接口在账号列表读模型上临时做 `HGETALL`、全量排序或全量 Redis 扫描。


推荐路径：

```text
GET  /api/v1/admin/scheduler-score/buckets
GET  /api/v1/admin/scheduler-score/buckets/:bucket/scores?page=1&page_size=50
POST /api/v1/admin/scheduler-score/buckets/:bucket/rebuild
```

接口行为：

1. `buckets` 返回已有 score bucket 的 meta。
2. `scores` 按 rank 分页返回某个 bucket 的账号分数。
3. `rebuild` 触发指定 bucket 重建。
4. `:bucket` 使用 `SchedulerBucket.String()`，注意 URL 编码。

如果要在第一阶段同时实现诊断接口，Redis 读模型必须额外增加 rank 索引，例如：

```text
scheduler_score:v1:{bucket:<bucket.String()>}:rank:<version>
```

其中 rank key 是 ZSET：

```text
member = account_id 十进制字符串
score = rank
```

诊断接口 `scores` 只能用 `ZRANGE rank:<version> start stop` 获取分页 accountIDs，再用 `HMGET scores:<version>` 读取详情。不允许用 `HGETALL scores:<version>` 后在内存排序分页。

诊断接口 `buckets` 也不能通过 `SCAN scheduler_score:v1:*` 枚举。若实现诊断接口，`SetBucketScores` 必须同时维护一个小型 bucket registry，例如：

```text
scheduler_score:v1:buckets
```

registry 可以使用 Redis Hash 或 ZSET 保存 `bucket.String()`、active version、bucket_size、updated_at。`DeleteBucket` 必须同步移除 registry 中对应 bucket。

## 第二阶段：账号调度选择器热路径优化

第二阶段是独立架构改造，目标是优化真实 OpenAI 账号调度选择器的请求热路径。第二阶段可以复用第一阶段补齐的 `SchedulerBucket`、调度快照重建链路和 active load map 能力，但不能复用 `scheduler_score:v1` 展示读模型作为真实选号依据。

必须先完成第一阶段并通过第一阶段验收，再进入第二阶段。除非负责人明确批准，第二阶段不得和第一阶段放在同一个提交或同一个上线批次中。

### 第二阶段目标

第二阶段分为两个可独立交付的小阶段。

#### 第二阶段 2A：活跃负载读取替换

目标是在真实 OpenAI 调度器中消除对完整候选池的 Redis 负载批查。

当前真实调度路径中，`defaultOpenAIAccountScheduler.selectByLoadBalance` 会对过滤后的候选账号构造 `loadReq`，然后调用：

```go
s.service.concurrencyService.GetAccountsLoadBatch(ctx, loadReq)
```

该调用对每个账号至少触发 `ZREMRANGEBYSCORE`、`ZCARD`、`GET` 三类 Redis 命令。大 bucket 下会把一次网关请求放大为数万到数十万 Redis 命令。

2A 完成后，真实调度器仍然可以扫描 bucket 内账号，但负载数据只能从 active load map 获取：

```go
s.service.concurrencyService.GetActiveAccountLoadMap(ctx)
```

active load map 只返回当前有并发槽位或等待队列的账号。候选账号不在 active load map 中时，必须按 0 并发、0 等待处理。

2A 的性能目标：

1. Redis 负载读取成本与活跃账号数相关，不再与 bucket 全量账号数相关。
2. 保持请求时完整过滤逻辑不变。
3. 保持 `AcquireAccountSlot` 原子抢槽逻辑不变。
4. 保持抢槽失败后的 fresh retry 兜底逻辑不变，但 fresh retry 只能读取小窗口候选，不允许重新读取完整候选池负载。

2A 的缓存注意事项：

- `GetActiveAccountLoadMap` 应加一个短 TTL 缓存（推荐 100-500ms），因为高频调度请求下每次 `ZRANGEBYSCORE now +inf` 读取整个活跃窗口本身可能成为瓶颈。
- 缓存的粒度是整个 active load map，而非按 accountID 独立缓存。
- 缓存击穿保护：使用 `singleflight` 或类似机制防止并发请求同时回源。
- fresh retry 路径可以绕过该缓存直接读取 Redis。
- 如果活跃账号数仍然很大（如 > 10000），必须继续坚持按 score 窗口读取，并优先优化 active index 清理和调用方缓存，不能退回 `ZRANGE 0 -1`。

#### 第二阶段 2B：候选索引分页读取

目标是在 2A 基础上，减少真实调度器请求时读取和过滤的账号数量。

2B 新增调度候选索引，按 `SchedulerBucket` 保存可分页读取的候选账号顺序。请求热路径按页读取候选窗口，逐页套用现有请求时过滤和抢槽逻辑。

2B 完成后，正常请求不应调用会一次性返回完整 bucket 的 `ListSchedulableAccounts`。只有以下情况允许回退旧路径：

1. 功能开关关闭。
2. 候选索引不存在或未就绪。
3. 候选索引读取失败。
4. 分页扫描达到 `selector_max_scan` 仍无法得到可用候选。
5. shadow compare 或灰度期间负责人要求强制回退。

2B 的性能目标：

1. 请求时账号读取数量与 `selector_candidate_page_size * page_count` 相关。
2. 正常情况下 `page_count` 应为 1 到 3。
3. 最终抢槽候选数量与 TopK 或小窗口相关。
4. 不再因单个 10 万级 bucket 导致网关请求读取 10 万账号快照。

以下行为一律不允许：

1. 不允许用 `scheduler_score:v1:*` 读模型直接决定真实账号选择。
2. 不允许删除或绕过 `tryAcquireAccountSlot` / `AcquireAccountSlot`。

4. 不允许删除 `isAccountRequestCompatible` 中任何现有约束。
5. 不允许删除 previous_response_id、session_hash、sticky weighted 的既有语义。
6. 不允许把请求时的 `requestedModel`、`requiredTransport`、`requiredCapability`、`requiredImageCapability`、`requireCompact` 只交给后台索引判断。
7. 不允许在请求热路径里出现全量 `ListAllWithFilters`、全量 DB 查询、全量 Redis pipeline。
8. 不允许在未开启 shadow compare 的情况下默认启用 2B。
9. 不允许因为候选索引缺失导致请求失败。索引缺失必须回退旧路径或返回原有错误。
10. 不允许用 goroutine 在请求内无边界异步重建候选索引。

### 第二阶段正确性原则

真实调度选择器分为三层，三层职责不能混淆：

1. 后台层：构建候选索引和读模型，只做可缓存、可重建、允许短暂陈旧的工作。
2. 请求层：根据本次请求上下文过滤候选，包括模型、能力、transport、compact、channel、quota、runtime block、excluded IDs、sticky 等。
3. 原子层：用 Redis 并发槽位做最终抢占，抢不到必须换候选或返回等待计划。

候选索引只能减少“看哪些账号”的成本，不能替代请求层和原子层。

5. `selector_candidate_page_size`

   - 候选索引不可用或扫描失败时是否回退旧选择器。

8. `selector_fresh_retry_window`
   - 默认 `64`。
   - 合法范围 `[8, 1024]`。
   - 首轮抢槽失败后，允许 fresh load retry 的候选窗口大小。
   - 不允许对完整 bucket 做 fresh retry。

#### Redis 实现要求

现有 Redis active index：

```text
concurrency:account:active_index
```

实现读取流程：

1. 使用 Redis `TIME` 获取 `now`。
2. 使用 `ZRANGEBYSCORE concurrency:account:active_index now +inf LIMIT offset activeAccountLoadReadChunkSize` 分块读取活跃 member。
3. 分页期间不删除 member，避免 `LIMIT offset` 因集合收缩跳过后续账号。解析失败、无真实负载的 member 先收集到 stale 列表，分页完成后批量 `ZREM`。
4. 分页变量要求：
   - `offset` 初始为 0。
   - 每次读取后 `offset += len(rawMembers)`。
   - 当 `len(rawMembers) < activeAccountLoadReadChunkSize` 时结束。
   - 返回 map 按 accountID 去重。
5. 每个 member 是 account id 十进制字符串。解析失败的 member 必须跳过并记录 warn。
6. 对活跃 member 分块读取真实负载。每个 member 可以复用现有内部逻辑读取：
   - account slot zset 的过期清理和 `ZCARD`
   - account wait key 的 `GET`
7. 每块大小常量：
   ```go
   const activeAccountLoadReadChunkSize = 500
   ```
8. 如果读取到 `slotCount == 0 && waitCount <= 0`，必须在分页完成后从 active index 删除该 member，避免索引残留。
9. 不允许一次 pipeline 超过 `activeAccountLoadReadChunkSize * 3` 个 Redis 命令。
10. 不允许在这个方法里读取数据库。
11. 不允许为了计算 `LoadRate` 反查账号并发配置。

注意：active index 是 best effort。遗漏活跃账号时，该账号会被当成 0 负载；最终 `AcquireAccountSlot` 会保证不会突破并发上限，但这不足以证明选择质量不变。2A 必须额外防止漏报繁忙账号占满 TopK 后把健康账号挤出 selection order；见下文“2A Active Index 漏报保护”。

helper 要求：

1. 只返回当前请求候选账号的负载，不返回 active map 中的无关账号。
2. 候选账号不在 active map 中时，返回 `CurrentConcurrency=0`、`WaitingCount=0`、`LoadRate=0`。
3. 候选账号在 active map 中时，复制 `CurrentConcurrency` 和 `WaitingCount`。
4. 使用 `account.EffectiveLoadFactor()` 计算 `LoadRate`。
5. 不允许修改 active map 中的原始对象，必须复制。

#### 2A Active Index 漏报保护

active index 是 best effort，不能把“最终抢槽不会超并发”误解为“选择质量一定不变”。如果活跃账号漏出 active map，首轮 load plan 会把这些账号当成 0 负载；在大 bucket 下它们可能占满 TopK，导致真实空闲但排序稍后的账号完全不进入 selection order。

2A 开启时必须满足以下任一保护策略，并用测试证明：

1. 首轮 active-load selection order 抢槽失败且候选池仍有未尝试账号时，按配置回退 legacy 负载读取，或扩大候选窗口后再尝试，不能直接返回 no available。
2. 如果选择扩大候选窗口，窗口上限必须有配置保护，且不能退回完整候选池 fresh retry。
3. no-available 前必须保留原有等待计划语义；不能因为 active map 漏报让可排队账号变成硬失败。
4. 测试场景必须覆盖：TopK 个账号在 Redis 中实际满载但 active map 漏报，TopK 之外存在健康账号；开启 2A 后最终仍能选中健康账号或按配置回退 legacy。

#### 2A Fresh Retry 限制

当前 `trySelectByLoadBalancePool` 抢槽失败后会调用：

```go
s.service.concurrencyService.GetAccountsLoadBatchFresh(ctx, loadReq)
```

2A 开启 active load map 后，不允许对完整 `filtered` 做 fresh retry。

要求：

1. 从当前 `attempt.selectionOrder` 中取前 `selector_fresh_retry_window` 个候选。
2. 只对这批候选调用 `GetAccountsLoadBatchFresh`。
3. fresh retry 重建 plan 时只能使用这批候选。
4. 如果这批候选仍无法抢槽，走原有等待计划或失败逻辑。
5. legacy 模式保持旧行为。

#### 2A 兼容性

2A 开关关闭时，`go test` 应证明真实调度选择器仍调用旧 `GetAccountsLoadBatch`。

2A 开关开启时，测试必须证明：

1. 首轮负载读取调用 `GetActiveAccountLoadMap`。
2. 首轮负载读取不调用 `GetAccountsLoadBatch`。
3. 缺失账号按 0 负载。
4. active map 中账号按 `EffectiveLoadFactor()` 重算 `LoadRate`。
5. fresh retry 只读取小窗口候选。
6. 最终仍调用 `AcquireAccountSlot`。
7. active map 漏报导致 TopK 候选满载时，不会直接把请求错误地变成 no available；必须触发保护策略。

#### Redis key 规则

新增文件：

```text
backend/internal/repository/scheduler_candidate_index_read_model.go
```

候选索引必须使用 active_version key：

```text
scheduler_candidate:v1:{bucket:<bucket.String()>}:version
scheduler_candidate:v1:{bucket:<bucket.String()>}:active
scheduler_candidate:v1:{bucket:<bucket.String()>}:ids:<version>
scheduler_candidate:v1:{bucket:<bucket.String()>}:accounts:<version>
scheduler_candidate:v1:{bucket:<bucket.String()>}:meta:<version>
```

key 说明：

1. `version` 是 string/integer counter，通过 `INCR` 分配单调版本。
2. `active` 是 string，值为当前 version。
3. `ids:<version>` 是 Redis ZSET。
   - member = account_id 十进制字符串。
   - score = rank，从 0 开始。
4. `accounts:<version>` 是 Redis Hash。
   - field = account_id 十进制字符串。
   - value = 白名单过滤后的 `service.Account` 调度元数据 JSON；不得包含真实上游凭据。
5. `meta:<version>` 是 Redis Hash。
   - `version`
   - `updated_at`
   - `bucket_size`

读取流程：

1. `GET active`。
2. 如果为空，返回 `Hit=false`。
3. `ZRANGE ids:<version> start start+limit-1`。
4. 对返回 ids 执行 `HMGET accounts:<version> ids...`。
5. 单条 JSON 反序列化失败时跳过该账号并记录 warn，不允许让整个请求失败。
6. 如果 `len(ids) == 0` 且 `start == 0`，返回 `Hit=true`、空页、`HasMore=false`。
7. `HasMore` 根据 `start + limit < bucket_size` 判断。

写入流程：

1. 调用方在 `SetBucketCandidates` 前调用 `NextVersion(ctx, bucket)`，通过 Redis `INCR` 获取单调版本；不得使用 `updatedAt.UnixNano()`。
2. 写入 `ids:<version>` 和 `accounts:<version>`。
3. 写入 `meta:<version>`。
4. 给新版本 ids/accounts/meta 设置临时 TTL，防止进程在 active 切换前崩溃留下未激活版本。
5. 使用同 slot Lua 原子切换 active：仅当 `new_version >= current_active_version` 时更新 active、`PERSIST` 新版本，并给旧版本设置宽限 TTL。
6. 如果 `new_version < current_active_version`，不得回滚 active；本次写入保持短 TTL 后自然清理。
7. 可异步清理旧版本，但不允许阻塞请求路径。

分块要求：

```go
const schedulerCandidateIndexWriteChunkSize = 1000
const schedulerCandidateIndexReadChunkSize = 512
```

不允许一次 pipeline 写入十万级命令。

#### 候选排序规则

后台写候选索引时，必须保持确定性排序。

第一版排序规则：

1. 分组内 `GroupPriority` 升序，`nil` 排在有值之后。
2. `Account.Priority` 升序。
3. `LastUsedAt == nil` 排在 `LastUsedAt != nil` 前。
4. `LastUsedAt` 升序。
5. `Account.ID` 升序。

说明：

1. 该排序只决定分页读取顺序，不是最终选号结果。
2. 最终选号仍由请求时评分、TopK、weighted selection 和 `AcquireAccountSlot` 决定。
3. 不允许在候选索引排序中加入当前负载，因为负载是实时状态。
4. 不允许在候选索引排序中加入 session_hash 或 previous_response_id，因为它们是请求上下文。

### 第二阶段 2B 选择器算法

新增方法：

```go
func (s *defaultOpenAIAccountScheduler) selectByLoadBalanceIndexed(
    ctx context.Context,
    req OpenAIAccountScheduleRequest,
) (*AccountSelectionResult, int, int, float64, error, bool)
```

最后一个 bool 表示是否成功使用 index 路径：

1. `true`：index 路径完成选择或返回确定错误。
2. `false`：index 不可用，调用方必须走 legacy。

主流程：

1. 根据 `req.GroupID` 和 `req.Platform` 构造 bucket：
   ```go
   SchedulerBucket{GroupID: normalizedGroupID, Platform: normalizeOpenAICompatiblePlatform(req.Platform), Mode: SchedulerModeSingle}
   ```
2. 从 `start=0` 开始分页读取候选。
3. 每页大小使用 `selector_candidate_page_size`。
4. 总扫描数不得超过 `selector_max_scan`。
5. 对每页账号逐个执行现有过滤：
   - `ExcludedIDs`
   - `IsSchedulable`
   - `Platform`
   - `IsOpenAICompatible`
   - `isOpenAIAccountRuntimeBlocked`
   - privacy group requirement
   - `isAccountRequestCompatible`
   - `isAccountTransportCompatible`
6. 把通过过滤的账号追加到 `filtered`。
7. 当 `len(filtered) >= selector_candidate_page_size` 或已无更多页时，使用当前 `filtered` 构建 load plan。
8. 使用 2A active load map 生成负载。
9. 调用现有 `trySelectByLoadBalancePool` 或等价的共享 helper。
10. 如果成功选中，返回。
11. 如果未选中但还有更多页，继续读取下一页。
12. 如果达到 `selector_max_scan` 仍未选中：
    - `selector_fallback_to_legacy_enabled=true`：返回 `usedIndex=false`，走旧路径。
    - `selector_fallback_to_legacy_enabled=false`：返回原有 no available 错误。

必须保留的行为：

1. `SubscriptionPriority` 仍然先分订阅账号池和常规账号池。
2. `RequireCompact` 仍然按 supported、unknown、stale snapshot retry 的顺序处理。
3. sticky weighted 仍然可以把 sticky 候选排到当前 TopK 前面。
4. `PreviousResponseCanMove=false` 时 previous response 仍然是硬粘连优先。
5. `session_hash` 非 weighted 模式仍然先走硬粘连。
6. 最终抢槽成功后仍然绑定 sticky session。

实现建议：

1. 先把现有 `selectByLoadBalance` 中“过滤账号”、“构建 load plan”、“订阅优先分池”、“fallback wait plan”拆成小 helper。
2. helper 必须保持原测试通过。
3. 再让 legacy path 和 indexed path 复用 helper。
4. 不允许为了复用 helper 改变原有错误类型和错误文案。

### Shadow Compare

2B 上线前必须实现 shadow compare。

shadow compare 运行方式：

1. 真实请求仍使用 legacy 选择器结果。
2. 按采样率并行或串行运行 indexed 选择器，但不能抢占真实并发槽。
3. shadow path 禁止调用 `AcquireAccountSlot`。
4. shadow path 只能计算候选顺序、候选数量、TopK、首个理论候选和失败原因。
5. shadow path 超时时间必须小于真实请求剩余 timeout，推荐 50ms 到 200ms。
6. shadow 失败只记录 debug/warn 指标，不影响真实请求。

必须记录的指标：

1. legacy selected account id。
2. indexed theoretical selected account id。
3. 两者是否一致。
4. legacy candidate count。
5. indexed scanned count。
6. indexed filtered count。
7. indexed page count。
8. indexed fallback reason。
9. indexed elapsed ms。
10. 是否命中 sticky / previous_response。

日志要求：

1. 默认不在 info 级别打印 account id。
2. debug 日志可以打印 hash 后的 request id 和少量 account id。
3. warn 只用于索引读取失败、反序列化异常、配置非法等需要运维关注的问题。

推荐日志名：

```go
slog.Debug("openai_selector_shadow_compare", ...)
slog.Warn("openai_selector_index_read_failed", ...)
slog.Warn("openai_selector_index_decode_failed", ...)
```

### 第二阶段上线流程

必须按顺序上线：

1. 发布包含 active load map 的版本，所有新开关保持 false。
2. 开启 `selector_active_load_map_enabled=true`，仅内部低流量灰度。
3. 观察 Redis command count、no available 错误、调度 P95/P99。
4. 2A 稳定后，发布候选索引代码，`selector_index_enabled=false`。
5. 开启候选索引后台重建，确认 index meta 和 bucket size 正常。
6. 开启 `selector_shadow_compare_enabled=true`，采样率 1%。
7. shadow compare 连续观察至少 24 小时。
